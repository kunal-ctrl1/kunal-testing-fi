FINORA — DETAILED RESUME CONTEXT (Frontend phase — transport proven, auth next)

PROJECT
Finora, an offline-first desktop accounting/billing app. Tauri v2 (Rust shell)
manages two child processes (sidecars): a Django/DRF backend (frozen to .exe via
Nuitka) and a portable PostgreSQL 18.4 instance. Frontend is SvelteKit (SPA mode,
adapter-static, ssr=false) + TypeScript + Svelte 5 (runes), talking to the local
Django server over a random localhost port. One user, multiple companies. Full
architecture and locked business decisions are in the two strategy files
(backend-strategy.md, frontend-strategy.md).

KEY DECISIONS THIS SESSION (so we don't relitigate)

- Transport = Tauri HTTP plugin (@tauri-apps/plugin-http), NOT webview fetch +
  django-cors-headers. Reason: webview loads from http://localhost:1420 (dev) /
  http://tauri.localhost (release), Django is on http://127.0.0.1:<port> — cross
  origin; plain webview fetch would fail CORS. Plugin fetch runs from Rust so CORS
  doesn't apply, behaves the same dev vs release, and gives a controllable cookie
  store for sessions. This avoids the cross-origin-credentialed-cookie-over-http mess.

- Chose the LAYERED path: prove bare transport (port handshake + splash + one real
  /health/ call) BEFORE adding auth, so auth debugging isn't tangled with transport
  debugging. Transport is now proven, so auth is unblocked.

- Auth strategy still OPEN. Backend currently uses DRF SessionAuthentication +
  IsAuthenticated (CSRF on writes, login flow needed). The plugin has a cookie store,
  but whether the session cookie PERSISTS across plugin-fetch requests is UNVERIFIED —
  that is the first thing to test in step 2. If session cookies don't persist cleanly,
  the fallback is to switch DRF to TokenAuthentication (Authorization: Token <key>
  header, no CSRF) — cleaner for desktop. Decide by testing, don't pre-build.
